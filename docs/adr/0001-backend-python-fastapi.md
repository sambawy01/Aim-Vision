# ADR-0001: Backend on Python 3.12 + FastAPI

**Status:** Accepted · **Date:** 2026-05-06 · **Owner:** Software Architect · **Supersedes:** the "Node.js or Python TBD" placeholder in `AIMVISION_V1_Sprint_Build_Plan.txt` v1.0.

## Context

The V1 sprint plan left the backend language explicitly undecided ("Node.js or Python TBD"). Every other architectural decision downstream — the ORM, the queue runtime, the async I/O posture, the CI toolchain, the SRE skill matrix, and the boundary with the ML training pipeline — depends on this choice. Leaving it unresolved past Sprint 3 forces every adjacent component to plan for both, doubling design surface for no leverage.

The shape of the work is unambiguous when laid out:

- **ML training pipeline** is Python-native. PyTorch, MMPose (RTMPose-x), VideoMAE-v2 backbones, ONNX export tooling (`onnx`, `onnxruntime`, `coremltools`), MLflow registry client, evaluation harnesses (`torchmetrics`, `netcal` for ECE), and the active-learning loop are all Python. See [AI Engineer review §Critical gaps and §Accuracy improvements](../reviews/01-ai-engineer.md).
- **Post-session inference workers** consume the same models the training pipeline produces. Round-tripping tensors through a Node service to call a Python inference service is one network hop and two serialization hops we do not need.
- **LLM coaching report generation** uses Ollama for self-hosted inference plus a fallback to Anthropic/Together (per [Performance review](../reviews/10-performance-benchmarker.md)). Structured output via `outlines` or `instructor`, RAG over per-athlete coach notes, and a verifier pass over the LLM output are all Python ecosystem.
- **Backend control plane** is mostly CRUD-with-business-rules: organizations, athletes, sessions, shots, recordings, subscriptions, federation appliances. Async I/O against Postgres, Redis, and S3 dominates. Pydantic validation parity with the TypeScript frontend (via `openapi-typescript` codegen) is significantly cleaner than maintaining two hand-written schemas.

The candidates considered were Python+FastAPI, Node.js+Fastify/Hono, Go+Echo, and Rust+Axum. Splitting languages across services (Node for control plane, Python for ML workers) was the most attractive alternative because it isolates blast radius — but it doubles the toolchain, fragments the SRE on-call rotation, and reimplements every DTO twice. The cost is paid every sprint, not once.

## Decision

**The backend is Python 3.12 + FastAPI + SQLAlchemy 2.0 + Pydantic v2 + Alembic + Arq (Redis-backed queue) + Uvicorn behind Granian.** Specifically:

- **FastAPI** as the web framework. Native async, OpenAPI generation out of the box, dependency-injection-style auth, and tight integration with Pydantic v2.
- **SQLAlchemy 2.0** as the ORM, in async mode (`asyncpg` driver). The 2.0 typed query API plus `mypy` plugin gives us an end-to-end typed path from request → handler → row.
- **Pydantic v2** as the schema language. Pydantic v2's Rust-backed validation is fast enough that we do not need to bypass it on the hot path. The OpenAPI spec emitted by FastAPI is consumed by `openapi-typescript` to generate the frontend `types.ts` on every backend PR via a CI check.
- **Alembic** for migrations, autogenerated from SQLAlchemy declarative models. Every migration is reviewed by a human; autogeneration is a starting point, not the final commit.
- **Arq** as the Redis-backed task queue for short-running async jobs (signed URL issuance with side effects, webhook dispatch, notification fan-out). Long-running and durable workflows go to **Temporal** (ADR-0007), not to Arq.
- **Uvicorn behind Granian** as the runtime. Granian (Rust-based, Rust-PSGI/ASGI server, by the Emmett author) gives us better p99 tails than Gunicorn-Uvicorn and supports HTTP/2 and HTTP/3 cleanly. Uvicorn is the inner ASGI server; Granian is the supervising runtime. We also benchmark Hypercorn as a fallback in case Granian regresses on a release.
- **`ruff` + `mypy --strict` + `pytest` + `pytest-asyncio`** as the lint/type/test toolchain.

Node.js is rejected as the backend language. Go and Rust are rejected: Go's lack of a Pydantic equivalent forces hand-written DTOs; Rust on the control plane is overkill for CRUD and slows the team for no measurable wins.

## Consequences

**Easier:**

- One language across backend, ML training, ML inference workers, evaluation harness, the LLM verifier pass, the data-labeling helpers, and the Temporal worker pool. One CI matrix, one container base image (`python:3.12-slim` plus a CUDA variant for ML), one `requirements.txt`/`pyproject.toml` style.
- Pydantic v2 schemas as the single source of truth for the API. Frontend TypeScript types are generated, not maintained.
- Hiring is straightforward: backend engineers and ML engineers draw from the same pool.
- The ML model registry (MLflow), the `model_version` lineage, and the shadow-routing infrastructure live in the same codebase as the inference workers that consume them.

**Harder:**

- Async Python under load is not Node.js. We need to be disciplined about not blocking the event loop with sync DB calls, sync CPU work, or unbounded list comprehensions on hot paths. We address this with `pytest-asyncio` style enforcement and Granian's request-time histograms exported to OpenTelemetry.
- Cold-start times for Python containers are slower than Node. We mitigate with Uvicorn's preload-app posture and by keeping the import graph shallow on the request path.
- GIL-bound CPU work must be moved off-process. Heavy CPU work (PDF rendering, image processing, ONNX inference) runs in worker processes (Arq workers and Temporal activity workers) with `numpy`/`onnxruntime` releasing the GIL during native calls.

**Reversibility:** Medium. The data layer (Postgres + RLS, see ADR-0004) and the API surface (REST + WebSocket) are language-agnostic. Rewriting the FastAPI handlers in Go later would be expensive but bounded; the data model and the frontend contract survive.

## Alternatives Considered

1. **Node.js + Fastify or Hono + Drizzle ORM + Zod.** Strong async story, end-to-end TypeScript types without codegen, large hiring pool. Rejected because it forces a language boundary at the ML interface — every model call becomes RPC, every shared schema is reimplemented, and the team operates two CI toolchains. The [Software Architect review §2](../reviews/03-software-architect.md) makes the case explicitly: "the plan reads ML-heavy."
2. **Go + Echo or Chi + sqlc.** Excellent runtime profile, single static binary, easy on SRE. Rejected because there is no Pydantic equivalent in the Go ecosystem — every request schema is hand-written, every ML interop is RPC, and the typed-frontend story regresses.
3. **Rust + Axum + sqlx.** Best-in-class performance, but the team is small and Rust is already required for the camera core (ADR-0003). Doubling Rust surface to also cover the CRUD backend is a hiring and velocity tax with no compensating win.
4. **Split-language: Node.js control plane + Python ML services.** The most attractive alternative on paper. Rejected because the boundary between "control plane" and "ML service" is fluid in practice — the post-session pipeline reads from Postgres, writes to Postgres, and calls models. Drawing the boundary forces network hops where in-process calls would do.
