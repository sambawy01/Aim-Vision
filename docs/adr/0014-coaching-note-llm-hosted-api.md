# ADR-0014: Coaching-Note LLM = Hosted API (with PII Strip Before Egress)

**Status:** Accepted · **Date:** 2026-05-24 · **Owner:** ML lead

## Context

The post-session coaching-note generator is wired through
`aimvision-ml/src/aimvision_ml/llm/` (PR #78). It accepts a session's
structured diagnostic outputs and produces a JSON-Schema-conforming
coaching note (`docs/llm-coaching-notes-schema.md`). The current
plumbing supports two backends: a self-hosted Ollama provider running
DeepSeek 14B Q4 (per V2 sprint plan) and an HTTP provider stub.

ADR-0012 commits us to on-prem-first GA at the federation tier. The
naive reading is "on-prem = self-host the LLM too" — that's what the
V2 plan implied. In practice:

- DeepSeek 14B Q4 needs ~14 GB VRAM. The federation appliance spec
  (one node, mid-range GPU at most) cannot host the model and run
  the rest of the stack comfortably.
- Self-hosting fine-tuning, eval gates, prompt-version rollouts, and
  hallucination QA *per federation* is operationally far heavier
  than centralizing it.
- The PII strip layer (`aimvision_ml/llm/pii.py`) already redacts
  athlete names, dates of birth, club / federation identifiers, and
  geo-location strings before the prompt body leaves the on-prem
  process. The LLM only ever sees role labels (`athlete_role`,
  `coach_role`) + structured diagnostic data + anonymous shot
  identifiers.

Hosted API options at production grade: Anthropic Claude, OpenAI
GPT-4o-class, Together AI (Llama-3.3 / DeepSeek-R1 hosted at the
weights level, not OpenAI-style abstraction). All three meet the
output-quality bar (we'll pick by eval gate + cost during Phase 1).

## Decision

**Coaching notes are generated by a hosted LLM API in Phase 1, with
mandatory PII strip + structured-output JSON-Schema enforcement
before egress, and audit-logged prompt/response pairs.**

Concretely:

- The default provider is set at Helm-chart deploy time:
  `LLM_PROVIDER=anthropic|openai|together`. The federation operator
  picks the one their data-residency posture accepts (Anthropic has
  an EU-region endpoint; Together hosts in US-East today).
- Before any prompt body leaves the on-prem process,
  `pii.strip_session_payload()` runs and the audit log records the
  pre-strip + post-strip diff hash. If `strict_egress=true` (default
  in production), any non-empty diff that contains an
  un-tokenised name / DOB / geo string aborts the call with a
  loud error rather than risking PII egress.
- The provider response is validated against the JSON Schema
  (`docs/llm-coaching-notes-schema.md`); validation failure
  re-prompts up to N times with the schema in-context, then falls
  back to a "coaching note unavailable" stub.
- An optional `LLM_PROVIDER=ollama` fallback continues to work for
  the in-house dev environment and for any federation that
  contractually forbids any cloud egress (they accept the lower
  quality + heavier ops as a trade-off).

## Consequences

**Positive.**
- Sidesteps the GPU-on-appliance hardware bill and the multi-week
  fine-tuning + inference-tuning ops work per federation.
- Higher-quality coaching notes than DeepSeek 14B Q4 at the same
  product-launch date.
- Centralized prompt versioning, eval gates, and rollouts.
- The PII strip layer is the same independent of provider, so the
  on-prem federation can audit a single egress boundary.

**Negative.**
- Internet egress from the on-prem appliance to a single allow-listed
  hostname is now load-bearing. The Helm chart documents the egress
  rule and the federation procurement pack discloses it.
- Per-token cost is now a variable operating expense. Phase-1
  budget: ≤ $0.50 per generated coaching note, achievable with
  current Anthropic Sonnet / GPT-4o-mini pricing for our ~3k-token
  prompt + 1k-token response.
- The hosted-API failure mode (network outage, provider rate-limit)
  becomes a user-visible failure mode for the coaching-note feature.
  Mitigation: the rest of the post-session pipeline (shot count,
  diagnostic chips) renders independent of the LLM call.
- A federation that contractually forbids cloud egress falls back to
  Ollama and accepts the quality + ops trade-off.

## Alternatives considered

- **Self-hosted DeepSeek 14B Q4 (V2 default)** — meets the on-prem
  purity criterion but raises hardware cost, ops surface, and
  per-federation tuning burden; quality is materially worse than
  hosted frontier models.
- **Self-hosted vLLM serving a smaller open model (Llama-3.1 8B)** —
  cheaper hardware but the quality drop on a structured-output
  coaching-note schema isn't acceptable for a coaching product.
- **No LLM-generated coaching note in Phase 1** — would ship the
  product without its headline feature; rejected.
